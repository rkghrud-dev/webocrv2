## 입력 우선순위

키워드 프롬프트는 아래 순서로 신뢰한다.

1. 원본 상품명과 GS코드
2. 원본 옵션 컬럼과 `optionItems`
3. OCR/사진 분석에서 상품과 직접 연결되는 사실
4. 이전 V파이프라인 키워드 후보

옵션, 가격, 상품코드처럼 구조화된 값은 OCR보다 항상 우선한다.
OCR은 상품 설명을 보강하는 자료일 뿐, 옵션/규격의 기준 데이터가 아니다.

## 상품명과 검색어 분리

상품명은 정확하게 간다.

- 대표어/표준어 우선
- 핵심 품목명이 앞쪽에 위치
- 기능/규격/용도는 1-2개만 반영
- 배송, 할인, 추천, 가격, 인기, 베스트, 핫템 같은 판매 문구 제외

검색어/태그는 넓게 간다.

- 표준어 + 현장명 + 별칭
- 상품명에 못 넣은 보조 키워드
- 실제 구매자가 검색할 가능성이 있는 말
- 무관 인기어, 일부러 만든 오타, 경쟁사 브랜드명 제외

## 옵션 처리

옵션값은 아래 순서로 둔다.

1. 옵션명/옵션값
2. 속성/상세정보
3. 검색태그/검색어
4. 상품명에는 대표값 또는 범위만

처리 기준:

| 구분 | 처리 |
|---|---|
| 단일 상품 | 상품 식별 규격을 상품명/검색어에 유지 가능 |
| 옵션 2-3개 | 필요 시 대표 옵션 1-3개 반영 가능 |
| 옵션 4-6개 | 범위 또는 선택형으로 압축 |
| 옵션 7개 이상 | 상품명에는 옵션값 나열 금지 |
| 색상 옵션 | 상품명에는 넣지 않고 옵션 컬럼에서 처리 |

옵션형 상품에서 숫자 규격은 OCR 숫자가 아니라 원본 옵션값을 기준으로 판단한다.
예를 들어 깔창 옵션이 `230 / 240 / 250 / 260 / 270`이면 `1763 mm` 같은 OCR 눈금 숫자는 버리고, 필요 시 `230-270 사이즈 선택형`처럼 옵션 범위만 쓴다.

이 규칙은 모든 마켓과 A/B 계정에 공통 적용한다.

금지:

- OCR에서만 나온 숫자 규격을 상품명에 넣기
- OCR 눈금, 이미지 배경 숫자, 측정 보조 숫자를 검색어로 쓰기
- 색상 옵션을 상품명에 넣기
- 옵션값 전체를 상품명 끝에 나열하기

허용:

- 원본 옵션값을 근거로 한 범위 압축
- 단일 상품의 식별 규격
- 재질명과 실제 상품 속성
- 상품명에 못 넣은 보조어를 태그/검색어로 분리

## OCR 잡음 제거 규칙

OCR에서 나온 아래 정보는 키워드 후보에서 제외한다.

- 배송, 급배송, 무료배송, 묶음배송, 상담, 문의, 전화번호, 주소

OCR에서 사용 가능한 정보는 상품 정체성, 소재, 기능, 사용처, 문제 해결, 실제 구성 정보뿐이다.

## A/B 변형 원칙

A/B 계정은 같은 상품을 올리지만 같은 상품명/태그를 그대로 복사하지 않는다.

- 가격은 동일하게 유지한다.
- 상품 정체성은 동일하게 유지한다.
- 옵션/색상은 옵션 컬럼에서 처리한다.
- A는 표준어/대표어 중심으로 만든다.
- B는 대체명/동의어/사용처 순서를 바꿔 만든다.
- 무관 키워드, 인기어, 오타 유도어는 A/B 모두 금지한다.
- 네이버는 40자 내외를 우선한다.
- 쿠팡은 네이버보다 조금 길게 가져간다.
- ESM/11번가는 짧고 압축된 엑셀 업로드형으로 만든다.
- 롯데ON은 네이버와 쿠팡 사이의 중간형으로 만든다.

## 현재 코드 반영

2026-05-13 현재 `components.jsx`의 임시 생성식은 위 기준에 맞춰 조정했다.

중요: 현재 WEBOCRV2의 키워드 생성 화면은 Codex/OpenAI를 다시 호출하지 않는다.
1차 시드에 저장된 OCR/사진분석/후보 풀을 프론트 규칙으로 조합하는 임시 생성식이다.
따라서 후보 풀이 약하면 상품명과 검색어도 약해진다.

목표 구현은 키워드 생성 단계에서 서버 작업을 다시 생성하고, 각 상품의 구조화 시드 데이터와 이 문서의 규칙을 AI 프롬프트로 전달해 10개 실제 판매 채널 상품명/검색어 패키지를 만드는 방식이다.

- 후보 풀을 그대로 나열하지 않고 상품명용 핵심어와 검색어/태그용 보조어로 분리한다.
- 네이버는 `스마트스토어 태그`, 쿠팡은 `쿠팡 검색태그`처럼 마켓별 라벨을 다르게 보여준다.
- 색상 옵션은 상품명에 넣지 않고 옵션 컬럼에서 처리한다. 규격/사이즈 옵션만 필요 시 `규격 선택형`, `N종 선택형`처럼 압축한다.
- 판매 문구와 공간 낭비형 키워드는 최종 후보에서 제외한다.
- 상품명에서는 붙여쓴 합성어를 풀어 쓴다. 예: `와이어릴고리` -> `와이어 릴고리`, `카라비너릴고리` -> `카라비너 릴고리`.
- 네이버 상품명은 40자 내외를 우선하고 50자를 넘기지 않는다. 너무 짧으면 핵심상품군, 대체명, 기능, 사용처를 추가해 40자에 가깝게 채운다.
- 쿠팡은 네이버보다 조금 길게 가져갈 수 있다.
- Cafe24는 판매용 출력보다 공통 후보 풀 역할이므로 가장 넓게 합친다.
- A/B 계정은 같은 상품군을 유지하되, A는 대표어 중심, B는 대체명/사용처 중심으로 상품명 순서를 다르게 만든다.

다음 단계에서 실제 서버 프롬프트/API로 붙일 때는 이 문서를 기준으로 `market_keywords.py` 스타일의 마켓별 패키지 생성을 WEBOCRV2 서버 쪽으로 옮긴다.
